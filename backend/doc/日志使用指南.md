# 日志使用指南

## 日志分级说明

本系统采用分级日志策略，将不同类型和级别的日志分离存储，便于问题定位和性能监控。

## 日志文件说明

### 1. 主日志文件
- **文件名**: `dify-app.log`
- **内容**: 所有INFO及以上级别的日志
- **保留**: 30天
- **大小限制**: 10GB

### 2. 错误日志文件
- **文件名**: `dify-app-error.log`
- **内容**: 仅ERROR级别日志
- **保留**: 30天
- **大小限制**: 5GB
- **用途**: 快速定位系统错误和异常

### 3. 警告日志文件
- **文件名**: `dify-app-warn.log`
- **内容**: 仅WARN级别日志
- **保留**: 30天
- **大小限制**: 5GB
- **用途**: 监控潜在问题和性能瓶颈

### 4. 性能监控日志
- **文件名**: `dify-app-performance.log`
- **内容**: RAG检索等关键性能指标
- **保留**: 7天
- **大小限制**: 1GB
- **用途**: 性能分析和优化

### 5. API调用日志
- **文件名**: `dify-app-api.log`
- **内容**: 外部API调用记录
- **保留**: 7天
- **大小限制**: 2GB
- **用途**: API调用统计和问题排查

### 6. SQL日志
- **文件名**: `dify-app-sql.log`
- **内容**: 数据库SQL语句
- **保留**: 7天
- **大小限制**: 1GB
- **用途**: SQL性能优化（开发环境）

## 日志级别使用规范

### DEBUG
- **用途**: 详细的调试信息
- **场景**: 开发调试、问题定位
- **示例**: 变量值、中间状态

```java
logger.debug("查询参数: knowledgeBaseId={}, topK={}", kbId, topK);
```

### INFO
- **用途**: 关键业务流程信息
- **场景**: 正常业务流程、状态变化
- **示例**: 用户操作、请求完成

```java
logger.info("知识库问答完成 - 知识库ID: {}, 问题: {}", kbId, question);
```

### WARN
- **用途**: 警告信息，但系统仍可继续运行
- **场景**: 使用默认值、降级处理、性能警告
- **示例**: 配置缺失、缓存未命中

```java
logger.warn("获取知识库配置失败，使用默认配置 - 知识库ID: {}", kbId, e);
```

### ERROR
- **用途**: 错误信息，需要关注和修复
- **场景**: 业务异常、系统错误
- **示例**: 服务调用失败、数据验证失败

```java
logger.error("知识库问答失败 - 知识库ID: {}", kbId, e);
```

## 异步日志配置

系统使用异步日志来提升性能：

- **队列大小**: 512
- **不阻塞**: 设置为false，高负载时丢弃日志
- **丢弃阈值**: 0，不丢弃任何日志

## 日志格式

### 控制台输出
```
2026-01-15 17:00:00.123 [http-nio-8080-exec-1] INFO  c.g.a.d.k.service.KnowledgeBaseQAService - 知识库问答完成
```

- **时间**: yyyy-MM-dd HH:mm:ss.SSS
- **线程**: 当前执行线程
- **级别**: 彩色高亮显示
- **Logger**: 类名（36个字符）
- **消息**: 日志内容

### 文件输出
```
2026-01-15 17:00:00.123 [http-nio-8080-exec-1] INFO  c.g.a.d.k.service.KnowledgeBaseQAService - 知识库问答完成
```
- **时间**: yyyy-MM-dd HH:mm:ss.SSS
- **线程**: 当前执行线程
- **级别**: INFO/WARN/ERROR
- **Logger**: 类名（36个字符）
- **消息**: 日志内容
- **异常**: 完整异常堆栈（%exception{full}）

## 性能优化建议

### 1. 异步日志
- 使用AsyncAppender避免I/O阻塞
- 队列大小根据并发量调整
- 生产环境设置neverBlock=false

### 2. 日志级别
- 生产环境: INFO
- 测试环境: DEBUG
- 开发环境: DEBUG
- SQL日志: 生产关闭，开发开启

### 3. 日志滚动策略
- 按日期滚动: daily
- 按大小滚动: 单文件最大100MB
- 保留策略: 根据重要性设置不同保留天数

## 问题排查指南

### 1. 系统错误
1. 查看 `dify-app-error.log`
2. 搜索关键词: ERROR, Exception
3. 查看完整堆栈信息
4. 定位具体业务代码

### 2. 性能问题
1. 查看 `dify-app-performance.log`
2. 分析关键操作耗时
3. 查看 `dify-app-warn.log` 警告信息
4. 检查API调用日志

### 3. 业务问题
1. 查看 `dify-app.log` 主日志
2. 使用grep搜索相关业务ID
3. 查看完整调用链
4. 关联错误日志

### 4. 数据库问题
1. 开启SQL日志（开发环境）
2. 查看 `dify-app-sql.log`
3. 分析慢查询
4. 检查连接池状态

## 日志最佳实践

### 1. 结构化日志
```java
// ✅ 推荐
logger.info("用户登录成功 - 用户ID: {}, 用户名: {}, IP: {}", userId, username, ip);

// ❌ 不推荐
logger.info("用户登录成功 userId=" + userId + " username=" + username + " ip=" + ip);
```

### 2. 异常日志
```java
// ✅ 推荐
logger.error("知识库问答失败 - 知识库ID: {}", kbId, e);

// ❌ 不推荐
logger.error("知识库问答失败 - 知识库ID: " + kbId);
logger.error("异常: ", e); // 分离记录，难以关联
```

### 3. 性能日志
```java
// ✅ 推荐
long startTime = System.currentTimeMillis();
// ... 业务逻辑 ...
long endTime = System.currentTimeMillis();
logger.info("RAG检索完成 - 耗时: {}ms, 检索结果数: {}", 
    endTime - startTime, results.size());

// ❌ 不推荐
logger.info("RAG检索完成");
```

### 4. 条件日志
```java
// ✅ 推荐
if (logger.isDebugEnabled()) {
    logger.debug("详细调试信息: {}", detailInfo);
}

// ❌ 不推荐（频繁调用会浪费性能）
logger.debug("详细调试信息: {}", expensiveToString());
```

## 日志监控建议

### 1. 关键指标
- ERROR日志数量趋势
- WARN日志数量趋势
- 日志文件大小增长
- 异步日志队列堆积

### 2. 告警规则
- ERROR日志超过阈值（如10次/分钟）
- WARN日志超过阈值（如100次/分钟）
- 日志文件大小超过阈值（如每天超过1GB）
- 异步日志队列超过阈值（如400/512）

### 3. 日志分析工具
- ELK Stack (Elasticsearch + Logstash + Kibana)
- Splunk
- Graylog
- Loki + Grafana

## 常见问题

### Q1: 日志文件过大怎么办？
A: 调整maxHistory和totalSizeCap，或使用日志归档工具。

### Q2: 异步日志丢失怎么办？
A: 检查queueSize设置，适当增加队列大小。

### Q3: 如何临时开启DEBUG日志？
A: 使用Logback的JMX或HTTP接口动态调整日志级别。

### Q4: 生产环境是否需要保留DEBUG日志？
A: 不需要，DEBUG日志量太大，且可能包含敏感信息。

### Q5: 如何优化日志性能？
A: 
1. 使用异步日志
2. 合理设置日志级别
3. 避免频繁的DEBUG日志
4. 使用条件日志判断

## 总结

合理的日志配置和使用规范对于系统维护至关重要：

1. **分级存储**: 快速定位问题
2. **异步输出**: 提升系统性能
3. **结构化日志**: 便于分析和监控
4. **合理级别**: 平衡性能和可观测性
5. **定期清理**: 避免磁盘空间耗尽

遵循本指南，可以显著提升系统的可维护性和问题排查效率。
