# 统一异常处理使用指南

## 概述

本系统通过`@RestControllerAdvice`实现统一的全局异常处理，确保所有异常都能被正确捕获、记录和返回标准化的错误响应。

## 核心组件

### 1. GlobalExceptionHandler
全局异常处理器，负责捕获和处理所有异常。

**位置**: `com.github.app.dify.common.controller.GlobalExceptionHandler`

### 2. ErrorCode
错误码定义类，统一管理系统所有错误码。

**位置**: `com.github.app.dify.common.exception.ErrorCode`

### 3. 自定义异常类
- `BusinessException`: 业务异常
- `NotFoundException`: 资源未找到异常
- `UnauthorizedException`: 未授权异常
- `ForbiddenException`: 禁止访问异常

### 4. ApiResponse
统一响应格式，包含成功和失败的响应。

## 异常处理列表

### 1. 业务异常

**异常类型**: `BusinessException`

**HTTP状态码**: 400

**处理方法**: `handleBusinessException`

**使用场景**: 
- 用户名或密码错误
- Token无效或已过期
- 用户不存在
- 密码错误
- 验证码错误

**示例**:
```java
throw new BusinessException("用户名或密码错误", ErrorCode.LOGIN_FAILED);
```

### 2. 资源未找到异常

**异常类型**: `NotFoundException`

**HTTP状态码**: 404

**处理方法**: `handleNotFoundException`

**使用场景**:
- 知识库不存在
- 文档不存在
- 对话不存在
- 消息不存在

**示例**:
```java
throw new NotFoundException("知识库不存在", ErrorCode.KNOWLEDGE_BASE_NOT_FOUND);
```

### 3. 未授权异常

**异常类型**: `UnauthorizedException`

**HTTP状态码**: 401

**处理方法**: `handleUnauthorizedException`

**使用场景**:
- 用户未登录
- Token无效
- Token过期

**示例**:
```java
throw new UnauthorizedException("用户未登录", ErrorCode.UNAUTHORIZED);
```

### 4. 禁止访问异常

**异常类型**: `ForbiddenException`

**HTTP状态码**: 403

**处理方法**: `handleForbiddenException`

**使用场景**:
- 用户无权限访问资源
- 知识库访问权限不足

**示例**:
```java
throw new ForbiddenException("您没有权限访问该知识库", ErrorCode.FORBIDDEN);
```

### 5. 参数验证异常

**异常类型**: `MethodArgumentNotValidException`

**HTTP状态码**: 400

**处理方法**: `handleValidationException`

**使用场景**: 
- 使用`@Valid`或`@Validated`注解的参数验证失败

**示例**:
```java
@PostMapping("/user")
public ApiResponse<User> createUser(@Valid @RequestBody UserDTO userDTO) {
    // ...
}
```

**响应格式**:
```json
{
    "success": false,
    "message": "参数验证失败：username: 用户名不能为空；email: 邮箱格式不正确",
    "code": 400,
    "data": {
        "username": "用户名不能为空",
        "email": "邮箱格式不正确"
    }
}
```

### 6. 参数绑定异常

**异常类型**: `IllegalArgumentException`

**HTTP状态码**: 400

**处理方法**: `handleIllegalArgumentException`

**使用场景**:
- 参数类型转换失败
- 参数绑定失败

### 7. 缺少请求参数异常

**异常类型**: `MissingServletRequestParameterException`

**HTTP状态码**: 400

**处理方法**: `handleMissingServletRequestParameterException`

**使用场景**: 
- 缺少必需的`@RequestParam`参数

**示例**:
```java
@GetMapping("/user")
public User getUser(@RequestParam Long userId) {
    // 如果请求中没有userId参数，会抛出此异常
}
```

### 8. 参数类型不匹配异常

**异常类型**: `MethodArgumentTypeMismatchException`

**HTTP状态码**: 400

**处理方法**: `handleMethodArgumentTypeMismatchException`

**使用场景**:
- 请求参数类型与定义不符

**示例**:
```java
// 请求: /user?userId=abc
// 定义: @RequestParam Long userId
// 会抛出此异常
```

### 9. Spring Security访问拒绝异常

**异常类型**: `AccessDeniedException`

**HTTP状态码**: 403

**处理方法**: `handleAccessDeniedException`

**使用场景**:
- Spring Security权限检查失败
- 用户已登录但无权限访问

### 10. 请求方法不支持异常

**异常类型**: `HttpRequestMethodNotSupportedException`

**HTTP状态码**: 405

**处理方法**: `handleHttpRequestMethodNotSupportedException`

**使用场景**:
- 请求的HTTP方法不被支持

**示例**:
```java
// 定义: @GetMapping("/user")
// 请求: POST /user
// 会抛出此异常
```

**响应格式**:
```json
{
    "success": false,
    "message": "请求方法 'POST' 不支持，支持的请求方法: GET",
    "code": 405
}
```

### 11. 媒体类型不支持异常

**异常类型**: `HttpMediaTypeNotSupportedException`

**HTTP状态码**: 415

**处理方法**: `handleHttpMediaTypeNotSupportedException`

**使用场景**:
- 请求的Content-Type不被支持

**示例**:
```java
// 请求头: Content-Type: text/plain
// 期望: application/json
// 会抛出此异常
```

### 12. 请求体不可读异常

**异常类型**: `HttpMessageNotReadableException`

**HTTP状态码**: 400

**处理方法**: `handleHttpMessageNotReadableException`

**使用场景**:
- JSON格式错误
- 请求体为空但预期有内容

### 13. 路径变量缺失异常

**异常类型**: `MissingPathVariableException`

**HTTP状态码**: 400

**处理方法**: `handleMissingPathVariableException`

**使用场景**:
- 缺少必需的`@PathVariable`参数

### 14. 请求头缺失异常

**异常类型**: `MissingRequestHeaderException`

**HTTP状态码**: 400

**处理方法**: `handleMissingRequestHeaderException`

**使用场景**:
- 缺少必需的请求头

**示例**:
```java
@GetMapping("/user")
public User getUser(@RequestHeader("Authorization") String token) {
    // 如果请求头中没有Authorization，会抛出此异常
}
```

### 15. 绑定异常

**异常类型**: `BindException`

**HTTP状态码**: 400

**处理方法**: `handleBindException`

**使用场景**:
- 参数绑定失败
- 类似于`MethodArgumentNotValidException`但用于不同场景

### 16. 约束违反异常

**异常类型**: `ConstraintViolationException`

**HTTP状态码**: 400

**处理方法**: `handleConstraintViolationException`

**使用场景**:
- 使用`@Validated`注解的方法参数验证失败

**示例**:
```java
@GetMapping("/user")
public User getUser(@Validated @RequestParam @NotNull Long userId) {
    // 如果userId为null，会抛出此异常
}
```

### 17. 文件上传大小超限异常

**异常类型**: `MaxUploadSizeExceededException`

**HTTP状态码**: 413

**处理方法**: `handleMaxUploadSizeExceededException`

**使用场景**:
- 上传的文件超过配置的大小限制

**响应格式**:
```json
{
    "success": false,
    "message": "上传文件大小超过限制（最大 10MB）",
    "code": 3005
}
```

### 18. 无处理器异常（404）

**异常类型**: `NoHandlerFoundException`

**HTTP状态码**: 404

**处理方法**: `handleNoHandlerFoundException`

**使用场景**:
- 请求的URL没有对应的控制器

### 19. 异步请求超时异常

**异常类型**: `AsyncRequestTimeoutException`

**HTTP状态码**: 408

**处理方法**: `handleAsyncRequestTimeoutException`

**使用场景**:
- 异步请求处理超时

### 20. 运行时异常

**异常类型**: `RuntimeException`

**HTTP状态码**: 500

**处理方法**: `handleRuntimeException`

**使用场景**:
- 未捕获的运行时异常

### 21. 其他异常

**异常类型**: `Exception`

**HTTP状态码**: 500

**处理方法**: `handleException`

**使用场景**:
- 所有其他未捕获的异常
- 网络连接异常
- Socket超时异常

## 错误码规范

### HTTP标准错误码 (400-599)
- `400`: BAD_REQUEST - 请求参数错误
- `401`: UNAUTHORIZED - 未授权
- `403`: FORBIDDEN - 禁止访问
- `404`: NOT_FOUND - 资源未找到
- `408`: REQUEST_TIMEOUT - 请求超时
- `413`: PAYLOAD_TOO_LARGE - 请求体过大
- `429`: TOO_MANY_REQUESTS - 请求过于频繁
- `500`: INTERNAL_SERVER_ERROR - 服务器内部错误
- `503`: SERVICE_UNAVAILABLE - 服务不可用
- `504`: GATEWAY_TIMEOUT - 网关超时

### 业务错误码 (1000-1999)
- `1001`: LOGIN_FAILED - 用户名或密码错误
- `1002`: TOKEN_INVALID - Token无效或已过期
- `1003`: USER_NOT_FOUND - 用户不存在
- `1004`: USER_ALREADY_EXISTS - 用户已存在
- `1005`: PASSWORD_ERROR - 密码错误
- `1006`: OLD_PASSWORD_ERROR - 旧密码错误
- `1007`: PASSWORD_SAME_AS_OLD - 新密码不能与旧密码相同
- `1008`: CAPTCHA_ERROR - 验证码错误
- `1009`: CAPTCHA_EXPIRED - 验证码已过期

### 资源错误码 (2000-2999)
- `2001`: RESOURCE_NOT_FOUND - 资源不存在
- `2002`: RESOURCE_ALREADY_EXISTS - 资源已存在
- `2003`: RESOURCE_IN_USE - 资源被占用
- `2004`: RESOURCE_DELETED - 资源已删除

### 文件错误码 (3000-3999)
- `3001`: FILE_NOT_FOUND - 文件不存在
- `3002`: FILE_UPLOAD_FAILED - 文件上传失败
- `3003`: FILE_DOWNLOAD_FAILED - 文件下载失败
- `3004`: FILE_TYPE_NOT_SUPPORTED - 文件类型不支持
- `3005`: FILE_TOO_LARGE - 文件过大
- `3006`: FILE_ALREADY_EXISTS - 文件已存在
- `3007`: FILE_DELETE_FAILED - 文件删除失败

### AI相关错误码 (4000-4999)
- `4001`: API_CALL_FAILED - API调用失败
- `4002`: API_TIMEOUT - API超时
- `4003`: API_CONFIG_ERROR - API配置错误
- `4004`: MODEL_NOT_FOUND - 模型不存在
- `4005`: APP_NOT_FOUND - 应用不存在
- `4006`: KNOWLEDGE_BASE_NOT_FOUND - 知识库不存在
- `4007`: DOCUMENT_NOT_FOUND - 文档不存在
- `4008`: CONVERSATION_NOT_FOUND - 对话不存在
- `4009`: MESSAGE_NOT_FOUND - 消息不存在
- `4010`: APP_TYPE_MISMATCH - 应用类型不匹配
- `4011`: WORKFLOW_EXECUTION_FAILED - 工作流执行失败
- `4012`: MISSING_REQUIRED_INPUT - 缺少必需的输入参数

### 数据库错误码 (5000-5999)
- `5001`: DATABASE_ERROR - 数据库操作失败
- `5002`: DATABASE_CONNECTION_ERROR - 数据库连接失败
- `5003`: DATABASE_TIMEOUT - 数据库超时
- `5004`: DATA_ALREADY_EXISTS - 数据已存在
- `5005`: DATA_VALIDATION_FAILED - 数据验证失败

### 外部服务错误码 (6000-6999)
- `6001`: REDIS_CONNECTION_ERROR - Redis连接失败
- `6002`: MINIO_CONNECTION_ERROR - MinIO连接失败
- `6003`: ELASTICSEARCH_CONNECTION_ERROR - Elasticsearch连接失败
- `6004`: MCP_SERVICE_UNAVAILABLE - MCP服务不可用
- `6005`: EXTERNAL_SERVICE_TIMEOUT - 外部服务超时

### 系统错误码 (7000-7999)
- `7001`: SYSTEM_BUSY - 系统繁忙
- `7002`: SYSTEM_MAINTENANCE - 系统维护中
- `7003`: NOT_IMPLEMENTED - 功能未实现
- `7004`: CONFIG_ERROR - 配置错误
- `7005`: RATE_LIMIT_EXCEEDED - 限流

## 最佳实践

### 1. 使用自定义异常
```java
// ✅ 推荐 - 使用自定义异常
if (user == null) {
    throw new NotFoundException("用户不存在", ErrorCode.USER_NOT_FOUND);
}

// ❌ 不推荐 - 直接抛出RuntimeException
if (user == null) {
    throw new RuntimeException("用户不存在");
}
```

### 2. 提供有意义的错误信息
```java
// ✅ 推荐
throw new BusinessException("用户名或密码错误", ErrorCode.LOGIN_FAILED);

// ❌ 不推荐
throw new BusinessException("操作失败", ErrorCode.BAD_REQUEST);
```

### 3. 使用正确的错误码
```java
// ✅ 推荐
throw new NotFoundException("知识库不存在", ErrorCode.KNOWLEDGE_BASE_NOT_FOUND);

// ❌ 不推荐 - 使用通用错误码
throw new NotFoundException("知识库不存在", ErrorCode.NOT_FOUND);
```

### 4. 参数验证
```java
// ✅ 推荐 - 使用注解验证
public ApiResponse<User> createUser(@Valid @RequestBody UserDTO userDTO) {
    // ...
}

// ✅ 推荐 - 手动验证
public ApiResponse<User> createUser(@RequestBody UserDTO userDTO) {
    if (StringUtils.isBlank(userDTO.getUsername())) {
        throw new BusinessException("用户名不能为空", ErrorCode.BAD_REQUEST);
    }
    // ...
}
```

### 5. 异常日志
系统会自动记录所有异常日志，包含：
- 请求ID（如果有）
- 请求方法
- 请求URI
- 用户ID（如果有）
- 异常信息
- 异常堆栈

### 6. 敏感信息处理
```java
// ✅ 推荐 - 不暴露敏感信息
if (user == null) {
    throw new UnauthorizedException("用户未登录", ErrorCode.UNAUTHORIZED);
}

// ❌ 不推荐 - 暴露敏感信息
if (user == null) {
    throw new UnauthorizedException("用户ID: 123456 不存在", ErrorCode.UNAUTHORIZED);
}
```

## 日志上下文

异常处理器会自动记录以下信息到日志：

```
[RequestID=abc123] [Method=POST] [URI=/api/user] [UserID=789] - 业务异常: code=1001, message=用户名或密码错误
```

这有助于：
- 快速定位问题请求
- 追踪用户操作
- 分析异常模式

## 扩展异常处理

如果需要添加新的异常处理器：

```java
/**
 * 处理自定义异常
 */
@ExceptionHandler(CustomException.class)
public ResponseEntity<ApiResponse<Object>> handleCustomException(CustomException e) {
    logger.warn("自定义异常: {}", e.getMessage());
    return ResponseEntity.badRequest()
            .body(ApiResponse.error(e.getMessage(), ErrorCode.BAD_REQUEST));
}
```

## 注意事项

1. **异常层级**: 
   - 异常处理器按照从具体到一般的顺序匹配
   - 确保更具体的异常处理器在前面

2. **日志级别**:
   - 业务异常：WARN
   - 参数验证：WARN
   - 运行时异常：ERROR
   - 系统异常：ERROR

3. **用户友好**:
   - 对用户显示的错误信息应该简洁明了
   - 技术细节记录到日志
   - 避免暴露内部实现细节

4. **错误码使用**:
   - 优先使用已定义的错误码
   - 新增错误码时遵循规范
   - 保持错误码的语义一致性

5. **性能考虑**:
   - 异常处理不应该太耗时
   - 避免在异常处理器中执行复杂逻辑
   - 数据库操作应该在业务层完成

## 总结

统一异常处理提供了：
1. **标准化的错误响应格式**
2. **完整的异常覆盖**
3. **详细的日志记录**
4. **友好的错误提示**
5. **易于扩展的架构**

遵循本指南，可以确保系统异常处理的一致性和可维护性。
