# 异常处理优化说明文档

## 优化概述

本次优化主要针对异常信息泄露技术细节的问题进行了改进，通过完善的异常处理机制和错误码体系，确保系统安全性和用户体验。

## 优化前的问题

### 问题 1: 异常信息泄露技术细节
**现状**:
- 数据库异常直接返回 SQL 错误信息
- 堆栈信息可能暴露给前端
- 文件路径、包名等敏感信息泄露

**安全风险**:
- 暴露数据库表结构、字段名
- 暴露服务器文件路径
- 暴露技术栈信息（Spring、Hibernate等）
- 暴露配置信息（连接字符串等）

### 问题 2: 错误码不统一
**现状**:
- 错误码硬编码在各个地方
- 没有统一的错误码规范
- 前端难以根据错误码做统一处理

**维护问题**:
- 错误码重复定义
- 修改错误码需要多处修改
- 缺少错误码文档

## 优化方案

### 1. 完善全局异常处理器

**文件位置**: `backend/src/main/java/com/github/app/dify/common/controller/GlobalExceptionHandler.java`

#### 主要改进

##### 1.1 增强日志记录
```java
// 优化前
logger.error("运行时异常", e);

// 优化后
logger.error("运行时异常: {}", e.getMessage(), e);
logger.error("系统异常: {} - 类型: {}", e.getMessage(), e.getClass().getSimpleName(), e);
```

**改进点**:
- 记录完整的错误信息（用于排查问题）
- 记录异常类型（便于快速定位）
- 保留堆栈信息（仅日志，不返回给前端）

##### 1.2 添加消息脱敏方法

```java
/**
 * 清理和脱敏错误消息
 * 移除敏感的技术细节，返回用户友好的消息
 */
private String sanitizeErrorMessage(String originalMessage) {
    // 移除可能包含敏感信息的模式
    String[] sensitivePatterns = {
        // SQL 相关
        "SQL", "SQLState", "ErrorCode", "syntax error", "constraint",
        
        // 数据库连接信息
        "jdbc:", "postgresql://", "mysql://",
        
        // 堆栈信息
        "at ", "Caused by:", "Exception:", "java.lang.",
        
        // 文件路径信息
        "/", "\\", ".java", ".class",
        
        // 其他技术细节
        "classpath:", "jar:file:", "nested exception"
    };
    
    // 检测敏感信息，返回通用错误消息
    return "操作失败，请稍后重试或联系管理员";
}
```

**脱敏规则**:

| 敏感信息类型 | 检测模式 | 处理方式 |
|------------|---------|---------|
| SQL 错误 | SQL, syntax error, column, table | 替换为通用消息 |
| 数据库连接 | jdbc:, postgresql://, mysql:// | 替换为通用消息 |
| 堆栈信息 | at , Caused by:, java.lang. | 替换为通用消息 |
| 文件路径 | /, \\, .java, .class | 替换为通用消息 |
| 技术细节 | classpath:, jar:file: | 替换为通用消息 |

##### 1.3 技术性消息检测

```java
/**
 * 判断错误消息是否过于技术性
 */
private boolean isTechnicalMessage(String message) {
    int technicalIndicators = 0;
    
    String[] technicalPatterns = {
        "Exception", "Error", "failed", "timeout", "null",
        "not found", "illegal", "invalid", "constraint",
        "SQL", "database", "connection", "network"
    };
    
    for (String pattern : technicalPatterns) {
        if (message.toLowerCase().contains(pattern.toLowerCase())) {
            technicalIndicators++;
        }
    }
    
    // 如果包含3个或以上技术关键词，认为是技术性消息
    return technicalIndicators >= 3;
}
```

**检测逻辑**:
- 统计技术关键词数量
- 超过3个技术关键词视为技术性消息
- 自动替换为用户友好消息

### 2. 统一错误码定义

**文件位置**: `backend/src/main/java/com/github/app/dify/common/exception/ErrorCode.java`

#### 错误码分类

| 分类 | 范围 | 说明 |
|-----|------|------|
| 成功 | 200 | 请求成功 |
| 客户端错误 | 400-499 | 4xx HTTP状态码 |
| 服务器错误 | 500-599 | 5xx HTTP状态码 |
| 业务错误 | 1000-1999 | 用户、认证相关 |
| 资源错误 | 2000-2999 | 资源管理相关 |
| 文件错误 | 3000-3999 | 文件上传下载相关 |
| AI相关错误 | 4000-4999 | Dify API、模型、应用相关 |
| 数据库错误 | 5000-5999 | 数据库操作相关 |
| 外部服务错误 | 6000-6999 | Redis、MinIO、ES等 |
| 系统错误 | 7000-7999 | 系统级错误 |

#### 常用错误码

```java
// 用户相关
LOGIN_FAILED = 1001              // 登录失败

// 数据库相关

```

#### 工具方法


## 使用示例

### 1. 抛出业务异常

```java
// 使用预定义错误码
if (user == null) {
    throw new BusinessException(
        "用户不存在", 
        ErrorCode.USER_NOT_FOUND
    );
}

// 使用自定义错误码
if (password.length() < 6) {
    throw new BusinessException(
        "密码长度不能少于6位", 
        ErrorCode.BAD_REQUEST
    );
}

// 使用默认错误码（400）
if (!isValidToken(token)) {
    throw new BusinessException("Token无效");
}
```

### 2. 自定义异常类

```java
public class AppNotFoundException extends RuntimeException {
    public AppNotFoundException(Long appId) {
        super("应用不存在: " + appId);
    }
}
```

然后在异常处理器中添加处理：

```java
@ExceptionHandler(AppNotFoundException.class)
public ResponseEntity<ApiResponse<Object>> handleAppNotFoundException(AppNotFoundException e) {
    logger.warn("应用未找到: {}", e.getMessage());
    return ResponseEntity.status(404)
        .body(ApiResponse.error("应用不存在", ErrorCode.APP_NOT_FOUND));
}
```

### 3. 前端错误处理

```javascript
// 根据错误码处理
if (response.data.code === ErrorCode.USER_NOT_FOUND) {
    showToast('用户不存在，请重新登录');
    router.push('/login');
} else if (response.data.code === ErrorCode.API_TIMEOUT) {
    showToast('请求超时，请稍后重试');
} else if (ErrorCode.isClientError(response.data.code)) {
    showToast('请求参数错误');
} else if (ErrorCode.isServerError(response.data.code)) {
    showToast('服务器错误，请联系管理员');
}
```

## 最佳实践

### 1. 错误日志记录

```java

logger.error("用户登录失败: username={}, error={}", username, e.getMessage(), e);

// ❌ 避免：只记录消息，不记录堆栈
```

### 2. 异常消息编写

```java

// ❌ 避免：技术性消息

```

### 3. 错误码使用

```java

// ❌ 避免：硬编码错误码

```

### 4. 异常链传递


## 注意事项

### 1. 不要过度脱敏

```java
// ✅ 推荐：保留有用的业务信息
if (!isValidEmail(email)) {
    throw new BusinessException("邮箱格式不正确");
}

// ❌ 避免：过度脱敏，丢失有用信息
throw new BusinessException("操作失败");
```

### 2. 敏感操作审计

对于涉及安全、金钱的操作，即使返回通用错误消息，也要在日志中记录详细信息：
```java
try {
    paymentService.process(order);
} catch (PaymentException e) {
    logger.error("支付失败: order={}, amount={}, user={}, error={}", 
        order.getId(), order.getAmount(), order.getUserId(), e.getMessage(), e);
    throw new BusinessException("支付失败，请稍后重试");
}
```

### 3. 避免日志注入

清理日志中的用户输入，防止日志注入攻击：
```java
// ✅ 推荐：清理用户输入
String sanitizedInput = userInput.replaceAll("[\r\n]", "");
logger.info("用户输入: {}", sanitizedInput);

// ❌ 避免：直接记录用户输入
logger.info("用户输入: {}", userInput);
```
