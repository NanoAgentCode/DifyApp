# 后端可优化项

本文档记录已实施的优化与后续可选的改进建议，便于维护与演进时参考。

## 已实施的优化

### 认证与路由

- **JWT 拦截器注册**：在 `WebMvcConfig` 中注册 `JwtInterceptor`，对需鉴权接口统一解析 JWT 并设置用户信息；登录、注册、系统配置等公共接口已排除鉴权。
- **系统配置 API**：提供 `SystemConfigController`，支持按 key 获取配置值（`/api/system-config/value/{key}`）、按分组获取配置列表（`/api/system-config/group/{group}`），解决前端拉取系统配置时的 404 问题。

### 业务与缓存一致性

- **AI 应用删除**：`AiAppServiceImpl.deleteAiApp` 软删除时同步清除按 ID 与按 apikey 的缓存，避免已删除应用仍被查询到。
- **知识库文档变更**：在文档向量化成功、重索引、删除文档向量后，通过 `CacheMonitorService.evictRagCacheForKnowledgeBase(knowledgeBaseId)` 使 RAG 检索结果及时更新。
- **知识库向量配置变更**：当向量数据库 ID 或类型发生变化时，在 `KnowledgeBaseServiceImpl.updateKnowledgeBase` 中调用 `VectorStoreFactory.invalidateEmbeddingStore(knowledgeBaseId)`，保证使用最新向量存储配置。

### 线程安全与资源释放

- **VectorStore 策略**：`ElasticsearchVectorStoreStrategy`、`WeaviateVectorStoreStrategy`、`QdrantVectorStoreStrategy`、`ChromaVectorStoreStrategy`、`MilvusVectorStoreStrategy` 中用于客户端/配置缓存的 Map 已改为 `ConcurrentHashMap`。
- **Elasticsearch 客户端**：在 `ElasticsearchVectorStoreStrategy` 中，切换或重建客户端时对旧的 `ElasticsearchTransport` 显式关闭，避免连接泄漏。

### 代码与接口规范

- **BaseController**：新增 `getRole(HttpServletRequest request)`，在部分 Controller 中统一从请求属性获取角色。
- **PageResponse**：构造函数中对 `pageSize <= 0` 进行防护，按 1 处理，避免计算总页数时除零。
- **资源不存在处理**：使用 `ServiceHelper.checkNotNull` 在 `KnowledgeBaseController`、`DataSourceController`、`AiAppController` 等处统一处理“资源不存在”，替代手写 `if (resp == null) throw new NotFoundException(...)`。

## 后续可选优化建议

以下为可选改进，按需实施，避免过度优化。

- **分页参数上限**：对 `pageSize` 设置合理上限（如 100），防止单次查询过大。
- **业务异常层级**：业务校验与“资源不存在”尽量在 Service 层抛出，Controller 仅做参数校验与调用。
- **入参校验一致性**：关键接口统一使用 `@Valid` + DTO 约束（如 `@NotNull`、`@Size`），与现有校验方式保持一致。
- **日志与敏感信息**：生产环境避免在日志中输出完整 Token、密码；错误日志级别与内容便于排查又不过于冗长。
- **ServiceHelper 扩展**：其他模块在“查单条、不存在则抛 NotFound”的场景可复用 `ServiceHelper.checkNotNull`，保持风格统一。
