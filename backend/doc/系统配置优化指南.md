## 异步线程池配置

### 配置文件位置

`backend/src/main/java/com/github/app/dify/system/config/AsyncConfig.java`

### 当前配置

```java
private static final int CPU_COUNT = Runtime.getRuntime().availableProcessors();
private static final int CORE_POOL_SIZE = CPU_COUNT;        // 8核 = 8
private static final int MAX_POOL_SIZE = CPU_COUNT * 2;    // 8核 = 16
private static final int QUEUE_CAPACITY = 200;             // 队列容量
```

### 参数说明

| 参数 | 说明 | 计算公式 | 作用 |
|------|------|----------|------|
| `CORE_POOL_SIZE` | 核心线程数 | CPU核心数 | 常驻线程，处理正常任务 |
| `MAX_POOL_SIZE` | 最大线程数 | CPU核心数 × N | 高峰时创建的最大线程数 |
| `QUEUE_CAPACITY` | 队列容量 | 固定值 | 等待执行的任务队列 |

### 何时调整

#### 场景1：CPU密集型任务（向量化、数据分析）

**症状**：
- CPU使用率持续 > 80%
- 任务排队时间长
- 系统吞吐量低

**调整方案**：

```java
// CPU密集型任务：线程数 = CPU核心数
private static final int CORE_POOL_SIZE = CPU_COUNT;
private static final int MAX_POOL_SIZE = CPU_COUNT * 2;
private static final int QUEUE_CAPACITY = 500;  // 增大队列
```

**预期收益**：
- CPU利用率提升
- 任务排队时间减少

---

#### 场景2：IO密集型任务（文件上传、API调用）

**症状**：
- CPU使用率低 < 50%
- 大量任务等待
- 系统吞吐量低

**调整方案**：

```java
// IO密集型任务：线程数 = CPU核心数 × 4
private static final int CORE_POOL_SIZE = CPU_COUNT * 2;
private static final int MAX_POOL_SIZE = CPU_COUNT * 4;
private static final int QUEUE_CAPACITY = 1000;  // 增大队列
```

**预期收益**：
- 系统吞吐量提升 100-200%
- 等待时间减少 50%

---

#### 场景3：高并发短任务（API查询）

**症状**：
- 大量短时间任务
- 线程池频繁创建/销毁
- 系统响应慢

**调整方案**：

```java
// 高并发短任务：增加核心线程，减少队列
private static final int CORE_POOL_SIZE = CPU_COUNT * 2;
private static final int MAX_POOL_SIZE = CPU_COUNT * 4;
private static final int QUEUE_CAPACITY = 100;  // 减小队列
```

**预期收益**：
- 任务立即执行
- 响应时间降低

---

### 调整步骤

1. **修改配置**

```java
// AsyncConfig.java
@Override
public Executor getAsyncExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();

    // 根据场景调整
    executor.setCorePoolSize(CORE_POOL_SIZE);
    executor.setMaxPoolSize(MAX_POOL_SIZE);
    executor.setQueueCapacity(QUEUE_CAPACITY);

    executor.setKeepAliveSeconds(60);
    executor.setThreadNamePrefix("Async-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.setWaitForTasksToCompleteOnShutdown(true);
    executor.setAwaitTerminationSeconds(60);
    executor.initialize();

    return executor;
}
```

2. **重启应用**

```bash
# 停止应用
./stop.sh

# 启动应用
./start.sh
```

3. **验证配置**

```bash
# 查看日志，确认线程池配置
tail -f logs/application.log | grep "异步线程池已配置"
```

## 数据库连接池配置

### 配置文件位置

`backend/src/main/resources/application-hikari.yml`

### 当前配置

#### 开发环境

```yaml
maximum-pool-size: 10
minimum-idle: 2
```

#### 生产环境

```yaml
maximum-pool-size: 50
minimum-idle: 10
```

### 参数说明

| 参数 | 说明 | 推荐值 | 影响 |
|------|------|--------|------|
| `maximum-pool-size` | 最大连接数 | CPU数 × 2 + 有效磁盘数 | 同时执行的数据库查询数 |
| `minimum-idle` | 最小空闲连接 | maximum-pool-size / 5 | 保持的空闲连接数 |
| `connection-timeout` | 连接超时(毫秒) | 30000 | 获取连接的最大等待时间 |
| `idle-timeout` | 空闲超时(毫秒) | 600000 | 空闲连接存活时间 |
| `max-lifetime` | 最大生命周期(毫秒) | 1800000 | 连接最大存活时间 |

### 何时调整

#### 场景1：数据库连接超时频繁

**症状**：
```
Cannot get JDBC Connection
Connection timeout
```

**诊断**：

```sql
-- 查看当前数据库连接数
SELECT count(*) FROM pg_stat_activity;

-- 查看最大连接数
SHOW max_connections;
```

**调整方案**：

```yaml
# application-hikari.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 100    # 50 → 100
      minimum-idle: 20          # 10 → 20
      connection-timeout: 60000 # 30s → 60s
```

**预期收益**：
- 连接超时错误减少 90%
- 并发能力提升 100%

---

#### 场景2：数据库CPU使用率持续 > 80%

**症状**：
- 数据库服务器CPU高
- 查询响应慢
- 系统整体性能下降

**调整方案**：

```yaml
# 优化查询，减少连接数
spring:
  datasource:
    hikari:
      maximum-pool-size: 30    # 50 → 30（减少连接）
      minimum-idle: 10          # 保持不变
```

**同时优化查询**：
- 添加索引
- 优化慢查询
- 使用缓存

---

#### 场景3：高并发读操作

**症状**：
- 大量读操作（查询、列表）
- 数据库连接池耗尽
- 响应慢

**调整方案**：

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 200   # 大幅增加
      minimum-idle: 50          # 增加最小空闲数
      connection-timeout: 60000
```

**同时优化**：
- 使用读写分离
- 添加本地缓存
- 使用Redis缓存

## Tomcat线程池配置

### 配置文件位置

`backend/src/main/resources/application.yml`

### 当前配置

```yaml
server:
  port: 8080
  # Tomcat线程池使用默认配置
```

### 推荐配置

#### 开发环境

```yaml
server:
  port: 8080
  tomcat:
    threads:
      max: 200          # 最大线程数
      min-spare: 10      # 最小空闲线程数
    accept-count: 100    # 等待队列长度
    connection-timeout: 10000  # 连接超时（毫秒）
```

#### 生产环境

```yaml
server:
  port: 8080
  tomcat:
    threads:
      max: 800          # 最大线程数（根据负载调整）
      min-spare: 50      # 最小空闲线程数
    accept-count: 500    # 等待队列长度
    connection-timeout: 30000  # 连接超时30秒
    max-connections: 10000  # 最大连接数
```

### 参数说明

| 参数 | 说明 | 推荐值 | 影响 |
|------|------|--------|------|
| `threads.max` | 最大线程数 | 200 ~ 800 | 同时处理的HTTP请求数 |
| `threads.min-spare` | 最小空闲线程数 | max的10% | 常驻线程数 |
| `accept-count` | 等待队列长度 | 100 ~ 500 | 等待处理的请求数 |
| `connection-timeout` | 连接超时(毫秒) | 10000 | 连接建立超时时间 |
| `max-connections` | 最大连接数 | 10000 | 最大监听连接数 |

### 何时调整

#### 场景1：HTTP请求处理慢

**症状**：
- 大量请求等待
- 响应时间长
- 系统吞吐量低

**诊断**：

```bash
# 查看Tomcat线程状态
curl http://localhost:8080/actuator/metrics/tomcat.threads.busy
curl http://localhost:8080/actuator/metrics/tomcat.threads.config.max
```

**调整方案**：

```yaml
server:
  tomcat:
    threads:
      max: 400          # 200 → 400
      min-spare: 20      # 10 → 20
    accept-count: 200    # 100 → 200
```

**预期收益**：
- HTTP处理能力提升 100%
- 响应时间降低

---

#### 场景2：大量长连接（WebSocket、SSE）

**症状**：
- 连接数达到上限
- 新连接被拒绝

**调整方案**：

```yaml
server:
  tomcat:
    max-connections: 20000   # 10000 → 20000
    threads:
      max: 1000              # 增加线程数
      min-spare: 100          # 增加最小空闲数
    accept-count: 1000       # 增加等待队列
```

**预期收益**：
- 支持更多并发连接
- 长连接稳定性提升

#### 应用层监控

```java
// 添加线程池监控
@Bean
public ThreadPoolTaskExecutor asyncExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    // ... 配置 ...

    // 添加监控
    executor.setTaskDecorator(runnable -> {
        long start = System.currentTimeMillis();
        try {
            runnable.run();
        } finally {
            long duration = System.currentTimeMillis() - start;
            if (duration > 1000) {
                logger.warn("Slow async task: {}ms", duration);
            }
        }
    });

    executor.initialize();
    return executor;
}
```

**监控指标**：
- 活跃线程数

- 完成任务数

#### 性能拐点判断

**正常范围**：
- 响应时间随并发线性增长
- 错误率 < 1%

**性能拐点**：
- 响应时间急剧增加
- 错误率 > 5%
- CPU或内存使用率 > 80%

**优化策略**：
- 记录性能拐点的并发数
- 将实际并发控制在拐点的80%
- 预留20%安全空间

## 📞 常见问题

### Q1: 如何确定当前配置是否需要调整？

**A**: 观察以下指标：
- CPU使用率持续 > 80% → 增加线程池
- 数据库连接经常超时 → 增加连接池
- 响应时间持续增加 → 全面检查
- 错误率 > 1% → 需要优化

### Q2: 调整配置后系统崩溃怎么办？

**A**: 立即回滚：
1. 恢复原配置文件
2. 重启应用
3. 检查日志找出原因
4. 咨询技术团队

### Q3: 能否动态调整配置？

**A**: 部分可以：
- Spring Boot Actuator支持动态调整
- 部分参数需要重启应用
- 建议在低峰期调整

### Q4: 配置调整有什么风险？

**A**: 主要风险：
- 过度配置：资源浪费
- 不足配置：性能瓶颈
- 不当配置：系统不稳定

**建议**：
- 渐进式调整
- 充分测试
- 保留回滚方案
