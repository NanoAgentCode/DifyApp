# 数据库优化说明文档

## 优化概述

本次优化包含两个主要方面：
1. **数据库索引优化**：为高频查询字段添加索引，提升查询性能
2. **连接池配置优化**：配置 HikariCP 连接池，提升数据库连接管理效率

## 一、数据库索引优化

### 1. ChatMessage 表索引优化

**文件位置**: `backend/src/main/java/com/github/app/dify/chat/domain/ChatMessage.java`

**新增索引**:

```java
// 单列索引
```

| `idx_conversation_create_time` | `conversation_id`, `create_time` | 复杂查询优化 | 多条件查询 |
| `idx_model_create_time` | `model_id`, `create_time` | 模型使用统计 | 模型使用量统计 |

| `maximum-pool-size` | 20 | 10 | 50 | 最大连接数 |
| `minimum-idle` | 5 | 2 | 10 | 最小空闲连接数 |
| `connection-timeout` | 30000ms | 30000ms | 30000ms | 获取连接超时时间 |
| `idle-timeout` | 600000ms | 600000ms | 300000ms | 空闲连接超时 |
| `max-lifetime` | 1800000ms | 1800000ms | 3600000ms | 连接最大存活时间 |
| `connection-test-query` | SELECT 1 | SELECT 1 | SELECT 1 | 连接测试查询 |
| `leak-detection-threshold` | 60000ms | 30000ms | 0 | 连接泄漏检测 |
| `validation-timeout` | 5000ms | 5000ms | 5000ms | 验证超时时间 |

### 2. 环境差异配置

#### 开发环境
- **连接池大小**: 10（较小的连接池，节省资源）
- **最小空闲**: 2（保持少量空闲连接）
- **泄漏检测**: 30秒（便于发现问题）
- **适用场景**: 本地开发、单元测试

#### 生产环境
- **连接池大小**: 50（支持高并发）
- **最小空闲**: 10（保持足够连接应对突发流量）
- **泄漏检测**: 关闭（避免性能影响）
- **连接存活**: 1小时（减少连接回收频率）
- **空闲超时**: 5分钟（快速释放无用连接）
- **适用场景**: 生产环境、高并发场景

### 3. 连接池配置说明

#### maximum-pool-size（最大连接数）
- **作用**: 控制连接池中最大连接数量
- **计算公式**: `核心数 * 2 + 有效磁盘数`
- **调整建议**:
  - CPU 密集型：核心数 * 2
  - IO 密集型：核心数 * 2 + 有效磁盘数
  - 一般应用：20-50

#### minimum-idle（最小空闲连接数）
- **作用**: 保持最小数量的空闲连接，避免频繁创建/销毁
- **调整建议**:
  - 开发环境：2-5
  - 生产环境：10-20
  - 避免设置过大（浪费资源）

#### connection-timeout（连接超时）
- **作用**: 从连接池获取连接的最大等待时间
- **调整建议**:
  - 一般应用：30秒
  - 高并发应用：10-20秒
  - 超时后抛出异常，避免长时间阻塞

#### idle-timeout（空闲超时）
- **作用**: 空闲连接存活的最大时间
- **调整建议**:
  - 开发环境：10分钟
  - 生产环境：5分钟（快速释放）
  - 避免设置过短（频繁创建连接）

#### max-lifetime（连接最大存活时间）
- **作用**: 连接在池中的最大存活时间
- **调整建议**:
  - 开发环境：30分钟
  - 生产环境：1小时（减少回收）
  - 必须小于数据库的 `wait_timeout`

#### leak-detection-threshold（泄漏检测）
- **作用**: 检测连接是否未正确释放
- **调整建议**:
  - 开发环境：30秒（便于发现问题）
  - 生产环境：0（关闭，避免性能影响）
  - 建议在测试环境启用

## 四、数据库迁移

### 索引创建

由于使用了 JPA 的 `@Index` 注解，索引会在应用启动时自动创建。

**开发环境**（`ddl-auto: update`）:
- 自动创建新索引
- 自动更新表结构

**生产环境**（`ddl-auto: validate`）:
- 需要手动执行 DDL 语句
- 建议使用 Flyway 或 Liquibase 进行版本管理

### 手动创建索引（生产环境）

如果需要手动创建索引，可以执行以下 SQL：

```sql
-- ChatMessage 表索引
CREATE INDEX idx_conversation_id ON chat_message(conversation_id);
CREATE INDEX idx_conversation_sequence ON chat_message(conversation_id, sequence);
CREATE INDEX idx_create_time ON chat_message(create_time);
CREATE INDEX idx_conversation_create_time ON chat_message(conversation_id, create_time);
CREATE INDEX idx_model_create_time ON chat_message(model_id, create_time);

-- ChatConversation 表索引
CREATE INDEX idx_user_id ON chat_conversation(user_id);
CREATE INDEX idx_app_id ON chat_conversation(app_id);
CREATE INDEX idx_knowledge_base_id ON chat_conversation(knowledge_base_id);
CREATE INDEX idx_user_create_time ON chat_conversation(user_id, create_time);
CREATE INDEX idx_user_type ON chat_conversation(user_id, type);
```

## 五、监控和调优

### 1. 连接池监控

**推荐工具**:
- **Spring Boot Actuator**: 默认提供 `/actuator/metrics` 端点
- **HikariCP JMX**: 可通过 JMX 监控连接池状态
- **Prometheus + Grafana**: 可视化监控连接池指标

**关键指标**:
- `hikaricp.connections.active`: 活跃连接数
- `hikaricp.connections.idle`: 空闲连接数
- `hikaricp.connections.pending`: 等待连接数
- `hikaricp.connections.max`: 最大连接数
- `hikaricp.connections.min`: 最小连接数

### 2. 数据库查询监控

**启用慢查询日志**:

```sql
-- PostgreSQL
ALTER SYSTEM SET log_min_duration_statement = '1000';  -- 记录超过1秒的查询
SELECT pg_reload_conf();

-- MySQL
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 记录超过1秒的查询
```

**分析工具**:
- **PostgreSQL**: `EXPLAIN ANALYZE`
- **MySQL**: `EXPLAIN`
- **pgBadger**（PostgreSQL）或 **Percona PMM**（MySQL）

### 3. 性能调优建议

#### 根据监控数据调整连接池大小

**症状**:
- `connections.pending` 持续较高：增加 `maximum-pool-size`
- `connections.active` 长期接近最大值：增加 `maximum-pool-size`
- `connections.idle` 长期为0：增加 `minimum-idle`

**调整策略**:
```yaml
# 场景1：高并发，等待连接频繁
maximum-pool-size: 100
connection-timeout: 15000  # 减少超时时间

# 场景2：查询耗时短，连接空闲多
minimum-idle: 20
idle-timeout: 120000  # 增加空闲超时

# 场景3：连接泄漏
leak-detection-threshold: 30000  # 启用检测
```

#### 根据查询优化索引

**使用 EXPLAIN 分析查询**:
```sql
EXPLAIN ANALYZE 
SELECT * FROM chat_message 
WHERE conversation_id = 123 
ORDER BY sequence;
```

**检查索引使用情况**:
- 如果出现 `Seq Scan`（全表扫描）：考虑添加索引
- 如果 `Index Scan` 使用索引：索引有效
- 检查 `rows` 估算值是否准确

## 七、注意事项

### 1. 索引维护

- **定期重建索引**: 根据数据量定期 `REINDEX`
- **监控索引使用率**: 删除未使用的索引
- **避免过多索引**: 索引会占用存储并降低写入性能

### 2. 连接池配置

- **避免过度配置**: 过大的连接池会浪费资源
- **定期调优**: 根据实际负载调整配置
- **监控泄漏**: 及时发现和处理连接泄漏

### 3. 数据库配置

- **调整 `max_connections`**: 数据库最大连接数应大于应用连接池
- **配置连接超时**: 数据库 `wait_timeout` 应大于连接池 `max-lifetime`
- **优化缓冲池**: 根据服务器内存调整数据库缓冲区
