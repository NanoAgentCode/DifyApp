# 可观测性功能与使用指南

## 1. 目标与范围

本文档说明当前后端可观测性能力的设计、配置与使用方式，覆盖以下两类能力：

- LLM 调用追踪（请求、响应、Token、耗时、状态）
- 业务步骤追踪（step 级输入/输出/异常）

当前可观测性能力已接入以下主链路：

- `chat`（非流式 / 流式）
- `knowledgebase`（非流式 / 流式）
- `documentreader`（非流式 / 流式）
- `mcp`（浏览器检索、时间、位置）

## 2. 架构说明

### 2.1 模块职责

- `ops.trace`：统一追踪门面与模型，业务模块只依赖该模块
- `ops.observability`：查询控制器与历史兼容实现
- `ops.observability.adapter.TraceDocumentGatewayAdapter`：将 trace 存储端口适配到当前 ES 文档实现

### 2.2 调用链路

1. 业务代码调用 `TraceFacade.start(...)` 创建追踪句柄
2. 业务代码通过 `TraceStepCollector` 记录步骤级事件
3. 业务结束调用 `TraceFacade.success(...)`，异常调用 `TraceFacade.error(...)`
4. 存储层 `EsTraceStore` 异步写入 ES 文档
5. 管理查询通过 `/api/observability/...` 读取追踪数据

## 3. 数据模型

### 3.1 主追踪字段

- `traceId`：请求级链路 ID
- `spanId`：调用级链路 ID（通常与 ES 文档 ID 对应）
- `traceSource`：追踪来源（如 `chat_main`、`knowledge_base_qa`）
- `requestContent` / `responseContent` / `errorContent`
- `inputTokens` / `outputTokens` / `totalTokens`
- `createdAt` / `finishedAt` / `latency` / `status`

### 3.2 步骤字段（存于 `metaData.steps`）

- `stepCode`：步骤编码（建议全局唯一）
- `stepName`：步骤名称
- `status`：`SUCCESS` / `FAILED`
- `startAt` / `endAt` / `durationMs`
- `inputSummary` / `outputSummary` / `errorSummary`

### 3.3 异步一致性状态

若异步写入重试后仍失败，会在 `metaData.asyncState` 标记：

- `status = PARTIAL`
- `lastFailedOperation`
- `lastFailedAt`
- `lastError`

用于识别“追踪部分缺失”场景。

## 4. 配置项

### 4.1 追踪异步线程池

```yaml
trace:
  async:
    core-size: 2
    max-size: 8
    queue-capacity: 1000
```

### 4.2 可观测性总开关

```yaml
observability:
  enabled: true
```

### 4.3 ES 数据源

通过系统配置维护 `observability.elasticsearchDataSourceId`，用于指定追踪写入和查询使用的数据源。

## 5. 业务接入方式

### 5.1 基本接入（推荐）

```java
TraceHandle handle = traceFacade.start(startRequest);
TraceStepCollector collector = new TraceStepCollector(traceFacade, handle, traceSanitizer);

try {
    Result r = collector.trace(
        "STEP_CODE",
        "步骤名称",
        "输入摘要",
        () -> doBusiness(),
        out -> "输出摘要"
    );

    traceFacade.success(handle, "响应摘要");
    return r;
} catch (Exception e) {
    traceFacade.error(handle, e);
    throw e;
}
```

### 5.2 流式场景接入建议

- 在方法入口调用 `start(...)`
- 在 `doOnComplete` 调用 `success(...)`
- 在 `doOnError` / `onErrorResume` 调用 `error(...)`
- 对 `switchIfEmpty` 这类早返回分支补 `success(...)`

## 6. 查询与调试

### 6.1 追踪列表

- `GET /api/observability/traces`
- 常用参数：`model`、`provider`、`traceSource`、`conversationId`、`startTime`、`endTime`

### 6.2 追踪详情

- `GET /api/observability/traces/{id}`

### 6.3 步骤详情

- `GET /api/observability/traces/{id}/steps`

返回 `metaData.steps` 数组，可直接用于前端时间线展示。

## 7. 排障建议

- 查询不到追踪：
  - 检查 `observability.enabled`
  - 检查 `observability.elasticsearchDataSourceId`
  - 检查 ES 索引 `llm_traces` 是否可访问
- 有主记录但步骤缺失：
  - 查看 `metaData.asyncState` 是否为 `PARTIAL`
  - 检查 trace 线程池是否饱和（`queue-capacity` 过小）
- 流式链路未收口：
  - 检查 `doOnComplete` / `doOnError` / `switchIfEmpty` 是否都调用了 trace 收口

## 8. 开发规范

- 业务模块不得直接调用 ES 客户端
- 业务模块统一通过 `TraceFacade` + `TraceStepCollector` 接入
- 步骤摘要必须走脱敏（`TraceSanitizer`）
- 建议统一步骤编码命名：
  - `CHAT_*`
  - `KB_*`
  - `DR_*`
  - `MCP_*`
