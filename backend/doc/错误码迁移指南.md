# 错误码迁移指南

## 概述

本文档指导如何将系统中的异常迁移到统一的错误码体系。

## 已完成的修改

### 1. 基础设施层 ✅

#### 1.1 异常类修改
- **`BusinessException.java`**
  - 默认错误码从 `400` 改为 `ErrorCode.BAD_REQUEST`
  
- **`UnauthorizedException.java`**
  - 新增构造函数支持错误码参数
  - 默认使用 `ErrorCode.UNAUTHORIZED`
  
- **`NotFoundException.java`**
  - 继承自 `BusinessException`（原继承 `RuntimeException`）
  - 新增构造函数支持错误码参数
  - 默认使用 `ErrorCode.NOT_FOUND`

#### 1.2 错误码定义
- **`ErrorCode.java`**
  - 定义了完整的错误码体系（1000-7999）
  - 提供工具方法：`getHttpStatus()`, `isClientError()`, `isServerError()`

#### 1.3 全局异常处理器
- **`GlobalExceptionHandler.java`**
  - 增强：`sanitizeErrorMessage()` - 错误消息脱敏
  - 增强：`isTechnicalMessage()` - 技术性消息检测
  - 改进日志记录，保留完整错误信息

#### 1.4 控制器基类
- **`BaseController.java`**
  - `getUserId()` 方法使用 `ErrorCode.UNAUTHORIZED`

## 常见问题

### Q1: 保留原始异常时如何记录日志？

**A**: 使用带 cause 参数的构造函数

```java
try {
    // 业务逻辑
} catch (Exception e) {
    logger.error("操作失败: userId={}, error={}", userId, e.getMessage(), e);
    throw new BusinessException("操作失败", ErrorCode.API_CALL_FAILED, e);
}
```

### Q2: 什么时候用 `BusinessException`，什么时候用 `NotFoundException`？

**A**: 
- `NotFoundException`: 资源不存在（用户、模型、文档等）
- `BusinessException`: 其他业务逻辑错误（参数错误、验证失败等）

### Q3: 如何选择合适的错误码？

**A**: 参考 `ErrorCode.java` 中的注释和分类

```java
// 用户相关 → 1000-1999
ErrorCode.USER_NOT_FOUND
ErrorCode.LOGIN_FAILED

// AI相关 → 4000-4999
ErrorCode.MODEL_NOT_FOUND
ErrorCode.API_CALL_FAILED

// 资源相关 → 2000-2999
ErrorCode.RESOURCE_NOT_FOUND
ErrorCode.RESOURCE_DELETED

// 文件相关 → 3000-3999
ErrorCode.FILE_UPLOAD_FAILED
ErrorCode.FILE_TYPE_NOT_SUPPORTED

// 数据库相关 → 5000-5999
ErrorCode.DATABASE_ERROR
ErrorCode.DATABASE_CONNECTION_ERROR

// 外部服务相关 → 6000-6999
ErrorCode.EXTERNAL_SERVICE_TIMEOUT
ErrorCode.REDIS_CONNECTION_ERROR

// 系统相关 → 7000-7999
ErrorCode.SYSTEM_BUSY
ErrorCode.CONFIG_ERROR
```

### Q4: 异常消息应该多详细？

**A**: 
- 对用户：简洁明了，说明问题原因
- 对运维：通过日志记录详细信息

```java
// ✅ 推荐：用户友好的消息
throw new BusinessException("密码长度不能少于6位");

// ❌ 避免：过于技术性
throw new BusinessException("Password must be at least 6 characters");

// ❌ 避免：过于详细
throw new BusinessException("密码长度不能少于6位，密码必须包含大小写字母、数字和特殊字符");
```

### Q5: 如何处理未在 ErrorCode 中定义的错误？

**A**: 使用通用错误码或添加新的错误码

```java
// 使用通用错误码
throw new BusinessException("操作失败", ErrorCode.BAD_REQUEST);

// 或添加新的错误码到 ErrorCode.java
public static final int SPECIFIC_ERROR = 7999;
```

1. **安全性提升** - 统一的错误码体系
2. **可维护性提升** - 避免硬编码
3. **用户体验改善** - 友好的错误消息
4. **开发效率提升** - 清晰的错误分类

迁移完成后，系统将具备更完善的异常处理机制。
